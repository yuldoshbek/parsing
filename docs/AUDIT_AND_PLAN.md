## 1. Общая картина: что должно быть vs. что есть

| Компонент | Ожидание руководства | Текущее состояние |
|---|---|---|
| **Парсер** | Собирает отзывы **раздельно по SKU/цвету** | Собирает все отзывы из склейки целиком — **дублирует** одни и те же отзывы для разных вариантов |
| **Фильтр отзывов** | PRO-фильтр: убирает мусор (звёзды без текста + стоп-слова), **все негативные берём** | Фильтр по >17 слов применяется поверх парсинга — часть валидного негатива **теряется** |
| **% позитив/негатив** | В сумме должно давать **100%** от одной базы | Знаменатель неверный (сравнивают отфильтрованные с общим числом на ВБ) — сумма ~53%, **47% "пропадают"** |
| **Уровень 1 (кто/когда)** | Два отдельных блока: «кто» и «когда» | Смешаны в один уровень — **логические категории перепутаны** |
| **Рейтинг 3 звезды** | Относится к негативу (1-3 = негатив) | **Полностью выпадает** из расчёта — не учитывается ни в одном блоке |
| **QA-вопросы** | Должны прогоняться аналогично отзывам | Спарсены (1868 записей), но **аналитически не обработаны** |
| **Негатив в 4-5★ отзывах** | Должен быть выделен отдельным блоком | **Не реализован** — боли ищут только в 1-3★ |

## 2. Корневые баги: детальный разбор

### 🔴 БАГ #1 — «Склейка»: одни отзывы на разные SKU (КРИТИЧЕСКИЙ)

**Что происходит:**  
Парсер заходит на карточку по артикулу, но при переходе в раздел отзывов получает **все отзывы склейки**, а не отзывы конкретного варианта (цвет/тип пластика).

**Доказательство из транскрипции (14.04, ~15:00):**  
> «У этого и второго пример один и тот же. Это значит, что у них одна база отзывов... Что-то он не то делает. Ну, то есть он не разделяет по моделям.»

**Последствие:**  
- Отзывы на PCL-ручку присваиваются PLA-ручке и наоборот  
- Выводы об ошибках и болях некорректны для конкретного SKU  
- Данные для нескольких артикулов в `results/` идентичны (например, `480055076.json` и `480055077.json` имеют одинаковый размер — 17 032 байт)

**Что нужно сделать:**  
В `main.py` при загрузке отзывов нужно:  
1. **Вариант A** (предпочтительный): выбрать конкретный цвет/вариант в UI WB перед скачиванием отзывов  
2. **Вариант B**: скачать всё, затем фильтровать по признаку (цвет/тип пластика в тексте отзыва)

### 🔴 БАГ #2 — Неверный знаменатель в расчёте % позитив/негатив (КРИТИЧЕСКИЙ)

**Что происходит:**  
```
% позитивных = кол-во позитивных (из отфильтрованных) / кол-во ВСЕХ отзывов на ВБ
```
Вместо:
```
% позитивных = кол-во позитивных / кол-во ОТФИЛЬТРОВАННЫХ (рабочая база)
```

**Доказательство из транскрипции (14.04, ~25:00):**  
> «Он считает количество положительных отзывов из отфильтрованных, количество негативных из отфильтрованных, а делит на общее количество отзывов. Это неправильно. Делить нужно на количество отфильтрованных.»

**Последствие:**  
37% + 16% = 53% ≠ 100%. Метрика бессмысленна, вводит в заблуждение.

**Правильная формула фильтрационной воронки:**
```
Всего на ВБ:          156 отзывов
  └─ Звёзды без текста:  -72  (46%)
  └─ С текстом:           84
       └─ "Мусор" (стоп-слова, бесполезные): -20
       └─ [РАБОЧАЯ БАЗА]:  64 отзыва  ← знаменатель для % позитив/негатив
            └─ Позитивных (4-5★): X%
            └─ Нейтральных (3★):  Y%
            └─ Негативных (1-2★): Z%
            = 100%
```

### 🟠 БАГ #3 — Рейтинг 3★ выпадает из анализа (СРЕДНИЙ)

**Что происходит:**  
Таблица делит только на «позитив» (4-5★) и «негатив» (1-2★). Оценка 3★ **нигде не учитывается**.

**Доказательство (14.04, ~08:50):**  
> «А ещё есть оценка 3. Она, получается, вообще выпадает.»

**Решение:** Добавить 3★ в группу негатива (1-3★ = проблемные), либо выделить отдельным столбцом «нейтральные».

### 🟠 БАГ #4 — Уровень 1: смешение «кто» и «когда» (СРЕДНИЙ)

**Что происходит:**  
В одном блоке объединены категории разной природы:
- «кто»: ребёнку, дочери, сыну, внуку — **получатель**
- «когда»: Новый год, день рождения, 8 марта — **повод**

**Решение:** Разбить на два отдельных уровня:  
- **L1a. Кто покупает → кому:** родители→ребёнку, бабушки/дедушки→внукам, себе  
- **L1b. Когда/повод:** Новый год, ДР, 8 марта, 23 февраля, 1 сентября, прочее

### 🟡 БАГ #5 — Негатив в позитивных отзывах (4-5★) не собирается (НИЗКИЙ)

**Контекст (14.04, ~33:10):**  
> «Боль может быть как в негативных, так и в положительных отзывах. Надо посмотреть, какой негатив пишут в 4-5 звёзды. Там люди более адекватные, конструктивную обратную связь дают.»

**Решение:** Добавить отдельный блок в таблицу: «Конструктивный негатив из 4-5★» — семантический поиск по позитивным отзывам на маркеры недовольства.

Из просмотра `246290927.json` (Give Creative):

> [!WARNING]
> 100% вопросов без ответа — это либо баг парсера (не берёт ответы), либо продавец системно игнорирует покупателей. Это важный сигнал для анализа.

## 6. Ключевые инсайты по продукту (из данных)

> [!NOTE]  
> Это уже аналитика — предварительная, до перепарсинга. После исправления парсера цифры изменятся.

**Топ-боли рынка (из CSV отзывов + questions):**
1. **Несовместимость пластика** — «Подходит только PCL, другой нельзя» — пользователи не понимают ограничения
2. **Засорение сопла** — самая частая тема в вопросах: «в сопле остатки», «пластик комками», «не извлекается»
3. **Заряд и питание** — «беспроводная или нет?», «сколько заряжается?» — описание на карточке недостаточно
4. **Быстрая поломка** — «пару дней работала и сломалась», «на ДР подарили — не работает»
5. **Инструкция** — «как пользоваться инструментом для чистки?», «в инструкции нет информации»

**Топ-позитив:**
- Подарочная упаковка («красиво», «отличный подарок»)
- Безопасность («не обжигает», «не пахнет»)
- Лёгкость освоения ребёнком

## 7. Что НЕ пошло не так (правильная оценка)

> [!NOTE]  
> По словам Алексея (14.04): «Ты сделал ровно то, что мы обсуждали. Никуда не шёл. Это хорошо.»

Общее движение **в правильном направлении**. Проблема не в неверном понимании задачи, а в:
- Недостаточной **самопроверке результатов** перед показом (ошибки в данных, которые должны были быть замечены самим разработчиком)
- **Техническом ограничении парсера** — он не был изначально спроектирован под разделение по SKU
- **Отсутствии чётких критериев приёмки**: команда не зафиксировала заранее, что «% позитив + % негатив = 100% — обязательное условие»
