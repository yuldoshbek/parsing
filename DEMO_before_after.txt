╔══════════════════════════════════════════════════════════════════╗
║  КАК УСТРОЕНА «СКЛЕЙКА» НА WILDBERRIES                          ║
╚══════════════════════════════════════════════════════════════════╝

  На WB один товар может иметь несколько артикулов (nmId) —
  по одному на каждый вариант (цвет, комплектацию).

  Артикул 246290927 = Оранжевый  (110 оценок на карточке)
  Другой nmId склейки = FDM       (30 оценок на карточке)
  ──────────────────────────────────────────────
  Итого по двум карточкам:  140

  Но feedbacks API по root даёт:  151  ← расхождение 11 штук
  Причина: баг счётчика на стороне WB, не парсера.

  ВАЖНО: когда кликаешь «Отзывы» на карточке любого варианта —
  WB открывает ОДИН общий поток для всей склейки (все варианты
  перемешаны). Парсер разрезает их обратно по полю color в API.


╔══════════════════════════════════════════════════════════════════╗
║  ПРОБЛЕМА 1 — НЕВЕРНЫЙ total_wb В ПАПКЕ SKU                     ║
╚══════════════════════════════════════════════════════════════════╝

  Папка: results_test / 246290927_Оранжевый

  ДО (было):
  ──────────────────────────────────────────────
  ── ВОРОНКА ───────────────────────────────────
    Всего на WB:       151   ← НЕВЕРНО
                              (это счётчик ВСЕЙ склейки,
                               а не оранжевого артикула)
    Спарсено (год):    13
    Рабочая база:      12
    После PRO-фильтра: 8

  ПОСЛЕ (стало):
  ──────────────────────────────────────────────
  ── ВОРОНКА ───────────────────────────────────
    Оценок этого SKU на WB: 110   ← ВЕРНО
                                    (feedbacks nmId 246290927)
    Оценок всей склейки:    151   ← справочно
    Спарсено (год):          13
    Рабочая база:            12
    После PRO-фильтра:        8

  Что изменилось:
    card.wb.ru?nm=246290927 → feedbacks = 110  (per-nmId, точный)
    feedbacks API → feedbackCount = 151         (весь root)
    Раньше парсер смешивал их и брал 151 для всех папок.
    Теперь для папки Оранжевый → 110, для FDM папки (если запросить
    FDM nmId) → будет его собственный счётчик.


╔══════════════════════════════════════════════════════════════════╗
║  ПРОБЛЕМА 2 — ДУБЛИРОВАНИЕ ПРИ НЕСКОЛЬКИХ АРТИКУЛАХ СКЛЕЙКИ     ║
╚══════════════════════════════════════════════════════════════════╝

  Сценарий: в articles.txt указаны оба артикула одной склейки:
    246290927   (Оранжевый)
    246290927_FDM_nmId  (FDM)

  ДО (было):
  ──────────────────────────────────────────────
  Артикул 1 → GET feedbacks/v1/232715088 → 83 отзыва
  Артикул 2 → GET feedbacks/v1/232715088 → 83 отзыва (те же!)
  Результат: одни и те же отзывы в двух папках = дублирование данных

  ПОСЛЕ (стало):
  ──────────────────────────────────────────────
  Артикул 1 → GET feedbacks/v1/232715088 → 83 отзыва → кэш[root]
  Артикул 2 → root уже в кэше → API НЕ вызывается
              данные берутся из кэша, разрезаются по цвету
  Лог: "root=232715088 уже в кэше — повторный запрос к API пропущен"

  Плюсы: нет дублей, меньше нагрузки на WB API, быстрее.


╔══════════════════════════════════════════════════════════════════╗
║  ПРОБЛЕМА 3 — ЦВЕТ nmId НЕ ОПРЕДЕЛЯЛСЯ ИЗ CARD API             ║
╚══════════════════════════════════════════════════════════════════╝

  ДО (было):
  ──────────────────────────────────────────────
  get_product_info() возвращал: brand, name, total_wb, root
  Цвет конкретного nmId — не запрашивался и не сохранялся.
  Парсер не знал «какому цвету принадлежит этот артикул».

  ПОСЛЕ (стало):
  ──────────────────────────────────────────────
  card.wb.ru?nm=246290927 → colors[0].name = "Оранжевый"
  get_product_info() теперь возвращает: +nm_color, +total_wb_nm

  Лог при запуске:
  "nmId=246290927, root=232715088, nm_color='Оранжевый', total_wb_nm=110"

  Применение: при сохранении per-SKU папки, если color == nm_color,
  используется точный total_wb_nm (110), а не общий root (151).
